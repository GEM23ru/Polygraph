## 1. ЧТО ЭТО ЗА ПРОЕКТ

**Polygraph** — мульти-провайдерный AI-агент на Python. Цель пользователя:
сделать агента, который умеет то же, что ассистент Arena (поиск, чтение
веб-страниц, выполнение кода, работа с файлами, рассуждения), а в перспективе —
генерация изображений/видео, парсинг Telegram, инфографика для маркетплейсов.

Идея архитектуры: одно ядро (`core.py`) общается с 6 провайдерами LLM и
само маршрутизирует запрос на подходящую модель. Сверху — агентский слой
(`agent.py`) с инструментами (function calling).

## 3. ПРОВАЙДЕРЫ И КЛЮЧИ (КРИТИЧНО!)

Пользователь находится в **РОССИИ**. Это определяет всё:

| Провайдер | Из РФ без VPN | Free-лимит | Статус у пользователя |
|-----------|:---:|-----------|----------------------|
| OpenRouter | ✅ | ~50 запросов/день | ключ есть, лимит часто исчерпан |
| Google (Gemini) | ✅ | ~250/день | ключ есть, лимит часто исчерпан |
| **Groq** | ❌ **только VPN** | 14 400/день, очень быстрый | ключ есть, работает с VPN |
| Mistral | ✅ | free tier | ключа пока нет |
| Cohere | ✅ | ~1000/мес | ключа пока нет |
| Cerebras | ⚠️ часто блокирует РФ | — | ключа нет |

**ГЛАВНЫЙ ВЫВОД:** сейчас рабочая лошадка — **Groq + включённый VPN**.
Без VPN Groq отдаёт `Forbidden`. OpenRouter/Google работают без VPN,
но их дневной лимит легко исчерпывается тестами (429 = лимит, 503 = провайдер занят).

Бюджет пользователя: **только бесплатное** (платные API не используем).

## ФАЗА 4 (2026-06-09): YouTube-субтитры + улучшения PDF + analyze_pdf_page

**Триггер:** пользователь спросил про дешёвые способы работы с PDF/YouTube.
Объяснили: PDF уже через pypdf (текст, не Vision — в 10× дешевле), а YouTube
можно через `youtube-transcript-api` (субтитры вместо кадров — в 200+× дешевле).

**Что сделано (4 фичи):**

### 1) `watch_youtube` — новый инструмент
Читает субтитры YouTube-видео через `youtube-transcript-api` (бесплатно, без
ключей, MIT). Поддерживает:
- Любые форматы URL: `youtube.com/watch?v=`, `youtu.be/`, `/shorts/`, `/embed/`,
  просто video_id.
- Приоритет языков: пользовательский (по умолч. `ru`) → en → любой доступный.
- Ручные и авто-сгенерированные субтитры.
- Тайм-коды `[MM:SS]` или `[HH:MM:SS]` для длинных видео.
- Лимит 20К символов (~5000 токенов) — для часовых видео хватает.

Используется для: пересказов видео, конспектов лекций, ответов «на какой минуте
говорят про X», обзоров. Если у видео нет субтитров (приваты, live, редкие
случаи) — честное сообщение, не выдумка.

⚠️ Зависит от недокументированного внутреннего API YouTube — может временно
ломаться при изменениях со стороны Google (быстро фиксится сообществом).

### 2) `read_file` для PDF: лимит 25К + аргумент `pages`
- Лимит чтения: было 8К → стало **25К символов** (~15 стр. крупного шрифта).
- Новый аргумент `pages`:
  - `""` (по умолч.) → первые 30 страниц
  - `"1-10"` → диапазон
  - `"5"` → одна страница
  - `"last10"` → последние 10
  - `"all"` → все (для коротких PDF)
- В выводе каждая страница помечается `--- стр. N ---` (модель знает,
  откуда взяла факт).
- При обрезке — явная подсказка «всего стр. X, запрошены Y, для других
  вызови read_file(pages='Z-W')».
- Если PDF длиннее 30 стр., но pages не указан — в конце подсказка про навигацию.

### 3) `search_pdf` — новый инструмент
Поиск ключевого слова по ВСЕМУ PDF (включая страницы, не попавшие в read_file).
Возвращает фрагменты с номерами страниц. Полезно для длинных документов:
- «Найди в этом договоре пункт про неустойку» → `search_pdf('contract.pdf', 'неустойка')`
- Лимит: до 20 совпадений (чтобы ответ не раздулся).
- Контекст вокруг найденного: 200 символов по умолч. (можно изменить через arg).

### 4) `analyze_pdf_page` — Vision-fallback для сканированных PDF
Когда `read_file` вернул "[PDF без извлекаемого текста]" (PDF из картинок-сканов):
- Рендерит ОДНУ страницу в PNG через `pdf2image` (нужен системный poppler).
- Передаёт картинку в существующий `analyze_image` (Gemini Vision).
- ⚠️ Дорого (~1000 токенов на страницу) — используется только когда обычный
  read_file не сработал. В описании инструмента явно сказано «только для отдельных
  страниц сканов, не для целых документов».

Если poppler не установлен — честное сообщение с подсказкой «сделай скриншот
страницы вручную и пришли как картинку».

## ШАГ 3 ФАЗЫ 3 (2026-06-09): Gemini-ветка с prior_messages + фикс W1

**Триггер:** после Шага 2 OpenAI-провайдеры (Mistral/Groq/OpenRouter) уже видят
честную историю с tool_calls, но Gemini — нет. Если на втором сообщении была
картинка → Gemini ответил → на третьем вопросе текст идёт в Groq, но если снова
картинка → опять Gemini, и он не помнит, что было в чате. Закрываем эту дыру.

**Что сделано:**

### 1) Новый метод `Agent._openai_msgs_to_gemini(messages)`
Конвертер OpenAI-формата (роли user/assistant/tool с tool_calls) в нативный
Gemini-формат (Content с role="user"/"model", parts с Part.from_text /
Part.from_function_call / Part.from_function_response). Правила:
- `system` → пропускается (идёт отдельно через `system_instruction`).
- `user` с непустым content → `Content(role="user", parts=[Part.from_text(...)])`.
- `assistant` с текстом и/или tool_calls → `Content(role="model", parts=[...])`.
  В parts: сначала текст (если есть), потом по одному Part.from_function_call
  на каждый tool_call.
- `tool` → `Content(role="user", parts=[Part.from_function_response(...)])`.
  Имя функции восстанавливается из карты `tool_call_id → name`, собранной из
  предыдущих assistant.tool_calls.

### 2) `_run_via_google()` принимает `prior_messages`
- Сигнатура: `_run_via_google(msg, max_turns=5, prior_messages=None)`.
- Если `prior_messages` передан → конвертируется в Gemini contents и добавляется
  ПЕРЕД текущим user-message. Модель видит свои прошлые вызовы analyze_image и т.п.
- Если конвертация упала (типы Gemini могут меняться между версиями) — продолжаем
  без истории, чтобы вообще не сломать ответ.

### 3) Фикс W1 из аудита: SYS_PROMPT → system_instruction
Раньше системный промпт шёл в `contents` как обычный текст (модель воспринимала
его как первый user-turn — смешивалось с диалогом, ухудшало следование правилам).
Теперь — в правильное поле `GenerateContentConfig.system_instruction`. Это:
- улучшает соблюдение правил (особенно красной линии по цифрам);
- экономит токены (system_instruction не повторяется в каждом сообщении истории).

### 4) Передача `prior_messages` во все вызовы `_run_via_google` в `run()`
Три места:
- Vision-приоритет для картинок (если в сообщении есть image_url/файл).
- `force_model == "gemini"`.
- Резерв после OpenAI-провайдеров.
Везде теперь идёт `prior_messages=prior_messages`.

**Тесты:** `test_step3.py` (6 кейсов, все зелёные):
8. Простой user-message → правильная структура.
9. Assistant с текстом → role="model", part.text.
10. Assistant tool_call + tool response → правильная сопоставка по id.
11. System пропускается.
12. Пустой user пропускается.
13. Assistant с текстом И tool_call одновременно → две части в одном Content.

**Итого тестов в проекте:** 13 (3 + 4 + 6), все зелёные.

**ЧТО ЭТО ПОЧИНИТ В РЕАЛЬНОМ ИСПОЛЬЗОВАНИИ:**
- ✅ После картинки (обработанной Gemini) на третьем сообщении: если опять прислать
  картинку или задать вопрос с vision-триггером, **Gemini видит** прошлый
  analyze_image и его результат. Не будет переспрашивать «пришли картинку ещё раз».
- ✅ Если пользователь зафиксировал Gemini через дропдаун — он держит весь диалог
  с памятью, не повторяет вопросы.
- ✅ Gemini лучше следует правилам промпта (system_instruction вместо склейки в user).

**Известные ограничения:**
- Если в истории были картинки (как реальные изображения), они в Gemini пока не
  пересылаются — только текст ответов analyze_image. Полная мультимодальная история
  потребует хранить байты картинок в чате. Это отдельная фича, не критично.

  только при первом ответе закреплённой модели).

### 3) Новый порядок моделей в `_FC_PROVIDERS` — «умный гибрид»
```
1. mistral-large-latest          ← топ-качество, БЕЗ VPN, первым
2. groq (от сильной к слабой):
   gpt-oss-120b → qwen3-32b → gpt-oss-20b → llama-3.3-70b (бредит, последний)
3. openrouter free (резерв при упавшем Groq или выключенном VPN):

4. mistral-small-latest          ← последний шанс без VPN
5. cerebras llama3.3-70b         ← часто блокирует РФ, самый последний
```
Логика: первый — Mistral-large (без VPN, топ); если у пользователя нет ключа

и переходим к Groq. Внутри каждого провайдера — от сильной к слабой.

### 4) `web.py api_chat` — собирает `prior_messages` из `chat["turns"]`
- Берёт последние `MAX_HISTORY=6` turn-ов и конкатенирует их `messages` в один
  плоский список — это и есть честная история.
- Передаёт в `agent.run(user_full, prior_messages=...)`.
- Фолбэк: если у чата нет `turns` (старый чат до Шага 1), всё ещё работает по-старому
  через склейку `fallback_context + user_full`.

**Тесты:** `test_step2.py` (4 новых кейса), `test_step1.py` (3 старых) — всего 7 зелёных:
4. prior_messages реально доходят до провайдера (видны system + 4 prior + текущий user).
5. `last_messages` содержит ТОЛЬКО текущий turn, не накапливает историю.
6. При падении всех моделей `session_model` сбрасывается.
7. При ответе запасной модели `session_model` обновляется на неё.

## ШАГ 1 ФАЗЫ 3 (2026-06-09): сбор честной истории диалога

**Триггер:** пользователь увидел, что модели «прыгают» между сообщениями в одном
чате (llama → gemini → gpt-oss), а на 3-м follow-up «сколько логистика?» gpt-oss-120b
выдумал «3% — это логистика», хотя на скриншоте 3% — это «Витрина экспресс (EDBS)»,
вообще не логистика. Корень: модель видит историю как СКЛЕЕННЫЙ ТЕКСТ user-сообщения,
а не как настоящий диалог с её прошлыми tool_calls. Поэтому она не помнит, что
analyze_image на скриншоте таблицы возвращал — и придумывает.

**Что сделано (только сбор, ничего не передаём):**
- В `Agent` добавлено поле `last_messages: list[dict]` — лог последнего turn-а
  в OpenAI-формате (роли user/assistant/tool, с `tool_calls`).
- `Agent.run()` сбрасывает `last_messages` на каждый вызов.
- `_run_via_openai_compat`: финальный ответ ассистента дописывается в `msgs`,
  затем `msgs` (без `system`-промпта) сохраняется в `last_messages`.
- `_run_via_google`: ведёт «зеркальный» лог `mirror_log` параллельно с нативным
  `contents`. Gemini-формат function_call → OpenAI-стиль tool_calls с id вида
  `gem_N_funcname`. По окончании сохраняется в `last_messages`.
- В `web.py`: `CHATS[cid]["turns"]` — новый параллельный формат истории
  (`{ts, user_msg, messages, model, images}`), копит честные turn-ы. Старый
  `messages` оставлен для UI без изменений. `load_chats()` мягко мигрирует
  старые чаты (добавляет пустой `turns`).
- `new_chat()` создаёт chat с обоими полями.

**Тесты:** `test_step1.py` (3 кейса, всё зелёное):
1. Простой ответ без инструментов → `[user, assistant]`.
2. Ответ с tool_call → `[user, assistant(tool_calls), tool(result), assistant(final)]`.
3. `last_messages` сбрасывается между `run()`-ами, не накапливается.

**ПОВЕДЕНИЕ НЕ ИЗМЕНИЛОСЬ:** turns копится, но никуда не передаётся.
Шаги 2-4 впереди: передать turns в Groq/OpenRouter (Шаг 2), потом в Gemini
(Шаг 3), потом убрать старую `context`-склейку и костыли `refers_to_file` (Шаг 4).

**Проверка на стороне юзера:** ничего не должно измениться. Прежние галлюцинации
